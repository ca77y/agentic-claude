# Agents improvements

Append-only notes on friction in the pipeline itself. One `###` entry per proposed improvement.

### The coder's QA step has no carve-out for artifact-only tasks

- **Area**: `agent:coder`
- **Observed**: `plugins/ca77y-engineering/agents/coder.md` makes the QA handoff unconditional — step 2 requires "one scenario test per spec scenario" and step 3 says "Hand the result to the `ca77y-engineering:qa` subagent", with "Report up once qa is green" in the Rules. For a task whose deliverable is a Markdown agent definition in a repo with no test suite and no executable code (this repo — `plugins/*/agents/*.md` *is* the product), none of that applies: there is no test to write and no validation command for `qa` to run. The dispatching `lead` had to override the definition in its prompt ("there is no separate qa subagent step to hand this to since none of your tools apply here") and hand-restate the spec's own read-through validation. An override in a dispatch prompt is invisible to the next `lead`, so every artifact-only task pays the same cost, and a `coder` that follows its definition literally would dispatch `qa` into a repo with nothing to run.
- **Suggested change**: Add a conditional branch to the coder's loop: when the spec's deliverable is a non-code artifact and the project has no test runner, the spec's own stated validation procedure (here, a `Requirement: Validation is a criterion-by-criterion read-through`) *is* the QA step, performed by the `coder` and recorded in the report in place of a `qa` result. Phrase step 3 as "run the project's validation — via `ca77y-engineering:qa` where the project has one, or the spec's stated validation procedure where it does not" so the fallback is in the definition rather than in each dispatch prompt.

### The code-review skill is hardwired to GitHub PRs, but the reviewer's usual target is an uncommitted worktree

- **Area**: `skill:code-review`
- **Observed**: The `reviewer` definition says its target "is usually the **uncommitted working tree** — the pipeline commits only at PR time, so a build in progress has no committed diff to point at". The `code-review` skill it is told to invoke assumes the opposite at every step: step 1 checks whether "the pull request" is closed or a draft, step 7 re-checks PR eligibility, and step 8 requires posting the result with `gh` and linking every finding to `https://github.com/<owner>/<repo>/blob/<full-sha>/...#L1-L2`. On this task there was no PR, no commit on the branch, and (confirmed via `gh pr list`) no PR has ever been opened in this repo — so steps 1, 7 and 8 had to be dropped and the findings relayed as text instead. Every reviewer round in this pipeline pays that adaptation cost, and the "link every finding to a full-sha blob URL" rule cannot be satisfied at all for working-tree changes, since the lines under review are not in any commit.
- **Suggested change**: Give the skill an explicit working-tree mode: when the target is an uncommitted diff, skip the PR eligibility checks, cite findings as `path:line` against the working tree instead of blob URLs, and return the findings as the skill's result rather than posting them. Alternatively, have the `reviewer` definition state the substitution once so each round does not re-derive it.

### The code-review skill's 80-point confidence filter silently drops verified findings

- **Area**: `skill:code-review`
- **Observed**: The skill scores each candidate finding 0-100 and instructs "Filter out any issues with a score less than 80. If there are no issues that meet this criteria, do not proceed." That threshold is calibrated for auto-posting a public PR comment, where a false positive is expensive. In this pipeline the findings go privately to the `coder` for a fix round, where the cost of a false positive is one dismissal. On this task two findings scored 70 and 72 — both scorers verified them as real defects and confirmed they were on lines added by the diff; the scores were held down only because the defect originates in the spec's own wording and because the impact "depends on interpretation". Applied literally, the filter would have reported a clean pass and dropped both.
- **Suggested change**: Make the threshold depend on where the findings go — keep 80 for posting to a PR, and relay anything at or above ~50 when the result is returned to another agent, tagged with its confidence so the `coder` can weigh it. Failing that, the `reviewer` definition should say that the filter applies to PR comments only and that verified sub-threshold findings are still relayed with their score.

### The docs pass has no route for a durable doc a concurrent sibling story has locked

- **Area**: `agent:writer`
- **Observed**: `plugins/ca77y-engineering/agents/writer.md` docs pass steps 3–4 assume every doc the change owes can be written in this pass, and step 7 makes spec removal conditional on the `auditor` gate alone. On this task the project convention (`docs/CLAUDE.md`: "when an agent's behavior changes, update the README") owed a root `README.md` edit, but two sibling stories were editing that same file concurrently in their own worktrees, so the edit was deliberately deferred to a later shared PR. The spec's `Coordination` section was the only record of that deferral — and step 7 deletes the spec. Following the definition literally therefore destroys the record of an unmet documentation obligation and reports the docs tree as consistent. It survived here only because the spec named the deferral and the dispatching `lead` repeated it in the prompt; with either missing, the obligation vanishes silently at the moment the spec is removed.
- **Suggested change**: Add an explicit branch to the docs pass: when an owed durable doc cannot be written in this pass — a concurrent story holds the file, or the change belongs in a later shared PR — re-home the obligation into a durable artifact outside the spec before removing it (in this project, a follow-up story card on the board, which `docs/CLAUDE.md` already prescribes for an identified-but-undone change), and name that artifact in the final report. State the removal rule as: the spec may be removed once every obligation it carries has either been written into durable docs or re-homed this way, *and* the gate has passed — not on the gate alone.
